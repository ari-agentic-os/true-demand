# 💊 付费意愿清醒剂 · true-demand

> 帮 builder 测出真实付费意愿的 AI 技能 · Claude Code / Codex / 任意 AI 工具。

内测个个好评，一收费全员卸载？活跃度低、一谈付费就卡壳、不知道该加功能还是换方向？

本技能（内部名 `true-demand`）专治这种「自嗨式调研」——用真实的市场行为，测出用户账单里的真话，把你的注意力从「功能好不好用（可用性）」拉回最常被逃避的那条线：**商业可行性——用户究竟愿不愿意为此掏钱**。

不管你是**还没用户的 pre-MVP、有种子用户卡住的成长期，还是卖给团队的 B2B**，它都会先读你的形态，再自动切换打法。

## 🎬 1 秒体感：端到端使用案例 (End-to-End Example)

> 💡 许多初创项目死于「自嗨」，看看这个 Skill 是如何帮你快速「退烧」的：

```text
用户：
/开局 我做了一个面向独立创作者的"多平台排版优化与一键同步"的浏览器插件。在小红书和即刻发了内测，有 500 多个写小红书、公众号的博主留邮箱拿到了内测包。但现在发现大家下载后，大部分人只用它来"本地排版美化"，很少有人真正去点那个"一键同步到多平台"的核心功能（活跃度很低）。下个月我打算上线收费，每个月 15 块钱，不知道会不会直接死掉，怎么测？

AI（true-demand）：
拿到 500 个创作者的邮箱是一个经典的"操作成本错觉"。创作者们对"能让排版变好看"的工具有天然的尝鲜欲，留邮箱只是她们在对"写出爆款"做低成本的工具投资。现在她们"只排版、不同步"，说明你可能把一个低频的刚需（跨平台分发），做成了高频的附庸（本地美化工具）。

而且你其实已经握着最硬的证据：500 个人里大部分只本地排版、几乎没人点"一键同步"——这是真实行为数据（不是嘴上说的），它在喊：你可能把一个低频刚需（跨平台分发）做成了高频附庸（本地美化）。

别盲目开订阅。我在本地建了三个文件：
① 验证看板《看板_核心假设.md》——最大假设「创作者愿意每月掏 15 元省掉手动跨平台搬运的痛」，并预设了确认线 / 放弃线；
② 防客套访谈提纲《提纲_真需求访谈.md》——含一封"向核心作者请教工作流"的破冰邀约 + 5 个看穿客套话的行为追问；
③ 行动计划《行动计划_招募与实验.md》——怎么约到对的人，外加一个并行跑的行为实验：给"一键同步"挂个 fake-door / 预售（点了就弹"¥15/月解锁"），用真实点击和付款来测，而不是嘴上问。

这里有一个经典的"辨真话"提问示范：

❌ 不要问： "如果之后同步功能稳定了，一个月收 15 块钱，你愿意订阅支持一下吗？"（创作者会出于同行鼓励说"不贵、支持"，但付钱时立刻假装看不见）。

✅ 应该问： "上个月为了把内容发到不同平台，或者为了排版好看，你真正花过最贵的一笔钱或者时间成本是什么？（比如买过什么排版主页的 VIP、用过什么付费分发软件、或者花钱找过兼职助理）当时为什么愿意花这个钱？现在的工具帮你省掉这个成本了吗？"

⏸️ 第一步别贪多：从那 15 个"每周打开、却从不点同步"的人里**先约 1 个聊**（约成了再扩），同时把 fake-door 挂上去。给自己定个执行意图——比如"周三前发出 3 封邀约"。哪怕只聊到 1 个、只丢 1 条笔记，就能用 `/纠偏` 一起看她到底卡在哪。
```

**🔀 换个形态，它怎么拐弯**

上面是「成长期 × B2C」的样子。换个 builder，诊断会自动变：

- **还没用户（pre-MVP）：** 没有自己的用户可挖——它把"约种子用户"换成"去竞品社区 / 目标人群招募 + 一个冒烟测试（落地页真 CTA / 预售）"，问的是"这问题存不存在、你过去为它花过什么钱"，而不是问一个还没造出来的产品。
- **卖给团队（B2B）：** 使用者 ≠ 拍板人 ≠ 预算主，"公司会报销"不等于自掏腰包——实验从 fake-door 换成 **LOI / 付费试点 / 直接找预算 owner 谈审批路径**；样本更少，但每个都更重。

## 🛠️ 安装与快速开始

本技能是纯 Markdown（`SKILL.md` + `references/`），**同时兼容 [Claude Code](https://claude.com/claude-code) 与 Codex**，无需安装任何依赖即可运行。

### 方式一：Git Clone（推荐）

```bash
git clone https://github.com/ari-agentic-os/true-demand.git
```

把整个文件夹重命名为 `true-demand`，放进对应工具的 skills 目录：

| 工具 | 放置路径 |
|---|---|
| Claude Code | `~/.claude/skills/true-demand` |
| Codex（macOS / Linux） | `$CODEX_HOME/skills/true-demand`（默认 `~/.codex/skills/true-demand`） |
| Codex（Windows） | `C:\Users\<用户名>\.codex\skills\true-demand` |

放好后**重启一次会话**，技能即可被识别。

### 方式二：ZIP 下载

在 GitHub 点 `Code → Download ZIP`，解压后把文件夹重命名为 `true-demand`，放进上表对应目录，重启会话。

### 方式三：直接喂 Prompt（任意 AI 工具，零安装）

不想装也行：把 `SKILL.md` 全文复制进对话，开头加一句——「请扮演一个帮 builder（创始人 / 独立开发者 / 产品人）验证真实付费意愿的沉稳产品顾问，严格按以下规范工作」，再粘贴 `SKILL.md` 内容即可。

### 用起来

- **三个入口：** 中文敲 `/开局`、`/纠偏`、`/确诊`；英文敲 `/kickoff`、`/align`、`/diagnose`。你敲哪种语言，它就用哪种语言在本地落盘（绝不中英夹杂）。
  - **Claude Code：** 直接当原生指令用，或描述处境自动命中。
  - **Codex：** 用 `/skills` 菜单选 `true-demand`、用 `$true-demand` 提及、或直接描述你的处境（命中 description 自动隐式触发）；`/开局` 等动词照打也能命中。若没给出明确中/英动词，技能会先问一次用哪种语言。
- **去 AI 味（可选依赖）：** 若另外装了 `humanizer-zh`（中文）/ `tool-humanizer`（英文），产出落盘前会自动过一遍去 AI 味；没装也能正常用（技能会自己手工润色）。

## ⚡ 三大核心指令（中英镜像，文件持久化）

真实的需求验证往往要拖上几周、跨多次访谈，它抛弃了「在单一 Chat 窗口里硬耗」的模式，拆解为三个随时可单点触发、基于本地文件持久化的动作，并把每一轮串成闭环：

### 🚀 `/开局` · `/kickoff` —— 诊断现状与下发工具
- **什么时候用：** MVP 积攒了一批内测用户，但面临活跃度低、不愿付费或功能排期迷茫时。
- **它在干嘛：** ⓪ **先读你已有的东西**——repo 里有 PRD / 文档 / 原型**和已有数据（留存/工单/支付）**就先扫一遍，从文档抽隐含假设（真实数据则是最硬证据），只问没答的缺口，并判断你的形态（pre-MVP/成长期 · B2C/B2B）来调整方法；① **看穿表象风险**——把注意力从「可用性」稳稳拉回最常被逃避的「商业可行性」；② **本地落盘**——验证看板 `看板_核心假设.md` + 防客套访谈提纲 `提纲_真需求访谈.md` + **行动计划**（招募清单 + 一个行为实验）`行动计划_招募与实验.md`。
- **⏸️ 默认停点：** AI 在这里退场——**第一步要极小：先聊 1 个人**，同时并行跑行为实验（真实掏钱 > 口头意愿）；1 条笔记就能回来跑 `/纠偏`。确诊后会开下一轮，直到你有底气开发或果断止损。

### 🎯 `/纠偏` · `/align` —— 访谈复盘与心智翻译
- **什么时候用：** 调研途中，觉得「用户好像在说客套话」或「心里没底」时。
- **它在干嘛：** 当你的场外顾问。你把原始笔记/录音丢进来，它帮你客观分析潜台词——比如「只要公司报销我就买」，本质上是在逃避自费，预算主体根本不是他；或者帮你发现用户愿意花大价钱买别的东西，从而撬开新的付费点；并实时修正你下一场访谈的提问姿势，防止你被假阳性信号欺骗。

### 🏁 `/确诊` · `/diagnose` —— 跨文件综合与战略转向
- **什么时候用：** 一轮访谈聊到不再冒出新信息（按形态走，单类人群通常 6–12 个；定价 / PMF 还要更多）、手里攒了一堆笔记时。
- **它在干嘛：** 跨文件读取 `访谈笔记/` 下所有记录，用亲和图聚类真实付费信号与高价值痛点，结合机会-解决方案树（Torres）给一份《报告_终局确诊.md》：**做**（付费路径跑通，下发排期）·**转**（现有方向是伪需求，但发现底层刚需，教你复用技术资产优雅转向）·**弃**（彻底证伪，及时止损）。

完整运作流程见 [`docs/运作流程.zh.md`](./docs/运作流程.zh.md)。


## 🧠 底层方法论（不止访谈：吃数据 · 测真行为 · 闭环迭代）

口语是大白话，底层严格执行行业黄金验证标准——而且它不止「问」：还吃掉你已有的真实数据、并行跑真实行为实验、把每一轮串成闭环。（每条方法论名都链到原始出处）

| 验证维度 | 引入的方法论 | 在工作流中的卡点动作 |
|---|---|---|
| 复&#8288;用&#8288;已&#8288;有&#8288;数&#8288;据 | [证据阶梯](https://en.wikipedia.org/wiki/Lean_startup) + [数据三角校验](https://en.wikipedia.org/wiki/Triangulation_%28social_science%29) | 先扫 PRD/原型，并吃掉留存/漏斗/工单/支付等真实数据当最硬证据（行为阶梯 rung 5–6），答过的不再重问 |
| 商&#8288;业&#8288;红&#8288;线&#8288;卡&#8288;位 | [IDEO 三透镜模型](https://hbr.org/2008/06/design-thinking) | `/开局` 把 80% 权重锁定在商业可行性，先放下可用性虚荣 |
| 防&#8288;客&#8288;套&#8288;挖&#8288;动&#8288;机 | [The Mom Test](https://www.momtestbook.com/) · [JTBD 岗位任务理论](https://hbr.org/2016/09/know-your-customers-jobs-to-be-done) | 过滤未来意愿的客套话，只问过去真实行为；拆购买背后的社会/情感痛点与切换阻力 |
| 真&#8288;行&#8288;为&#8288;验&#8288;证 | [行为实验](https://en.wikipedia.org/wiki/Minimum_viable_product) + 证据阶梯 | 每轮访谈并行一个 pre-sell / fake-door / 付款实验，真实掏钱 > 口头意愿 |
| 形&#8288;态&#8288;分&#8288;诊 | [阶段 × 买家分诊](https://en.wikipedia.org/wiki/Customer_development) | pre-MVP/成长期 · B2C/B2B 各走不同打法；B2B 区分使用者 ≠ 拍板人 ≠ 预算主 |
| 信&#8288;号&#8288;收&#8288;敛&#8288;控&#8288;制 | [主题饱和](https://doi.org/10.1177/1525822X05279903) / [信息力](https://doi.org/10.1177/1049732315617444) | 聊到不出新主题就停（单 segment ≈6 起、~12 封顶）；尼尔森「5 人」只属可用性，不套到需求/定价 |
| 输&#8288;入&#8288;先&#8288;验&#8288;真 | 证据四分栏 + [推理阶梯](https://en.wikipedia.org/wiki/Ladder_of_Inference) | DID/SAID/COMMITTED/MY READ 拆开记；先审访谈者有没有诱导/pitch，行为 > 口头 |
| 定&#8288;价&#8288;策&#8288;略&#8288;破&#8288;局 | [Van Westendorp PSM](https://en.wikipedia.org/wiki/Van_Westendorp%27s_Price_Sensitivity_Meter) | 通过定性行为对比，推导出不流失内测粉的定价黄金带 |
| 解&#8288;构&#8288;功&#8288;能&#8288;迷&#8288;茫 | [Teresa Torres 机会解决方案树](https://www.producttalk.org/opportunity-solution-tree/) | 把杂乱笔记规整为「目标-机会-方案-实验」，理清功能排期 |
| 挽&#8288;救&#8288;活&#8288;跃&#8288;留&#8288;存 | [Superhuman 核心路径减法](https://review.firstround.com/how-superhuman-built-an-engine-to-find-product-market-fit/) | 通过超级用户反馈做功能减法，拯救低活跃度 |
| 技&#8288;术&#8288;资&#8288;产&#8288;复&#8288;用 | [最小技术资产留存转向（Pivot）](https://en.wikipedia.org/wiki/Lean_startup) | 证伪后不浪费一行老代码，复用现有技术资产匹配新的高客单价场景 |
| 预&#8288;设&#8288;决&#8288;策&#8288;线 | [确认线/放弃线](https://en.wikipedia.org/wiki/Preregistration_%28science%29) + 置信度 | 拿数据前就写死确认线/放弃线；按饱和 × 行为 × 跨人群给置信度，绝不靠口头证据拍板 BUILD |
| 极&#8288;小&#8288;启&#8288;动 | [执行意图](https://doi.org/10.1037/0003-066X.54.7.493) + [Tiny Habits](https://tinyhabits.com/) | 默认只约 1 个人 + 当场定「周三前发 3 封」，跨过 kickoff→首访的流失点 |
| 闭&#8288;环&#8288;迭&#8288;代 | [Build-Measure-Learn](https://en.wikipedia.org/wiki/Lean_startup) | `/确诊` 只关掉一轮——记进验证日志、点出下一个最大假设，再回 `/开局` |

设计原理与「怎么验证这个技能本身」见 [`ABOUT.md`](./ABOUT.md)。
